# 09. Designing AI-Native MVP Experiments, Not Features

> **Thesis.** Ship the smallest agent that proves the loop, not a pile of features. Your first agent should own one job and one job only: high-volume, guideline-based, low-risk. That is the Goldilocks zone, and starting anywhere else burns the trust you will need for everything that comes after.

## The shift

The cheapest thing in your company is now a feature. You describe one in a sentence and it works by lunch. That is the trap, dressed as a gift.

When building was slow, scope disciplined itself. Every extra screen cost engineering weeks, so you were forced to choose, and the choosing kept you honest. That brake is gone. A feature is a prompt and an overnight cycle, which means the price that used to stop you from building too much has fallen to roughly zero. What is left to stop you is judgement, and judgement is exactly what a build-anything machine quietly erodes.

So founders pile up features and call it a product. A dashboard with eight panels. A settings page. Three integrations and a roadmap of twelve more. It demos beautifully, and it answers none of the question that decides whether the company lives: will a real person let a machine own the one job the whole thing depends on. That hypothesis is the scary one, so it goes untested, while feature work goes on because feature work feels like progress.

The old target was a minimum viable product, a small pile of features arranged to look like a real thing. In an AI-native company that is the wrong target. You are not shipping a product made of features. You are shipping an intelligence that does a job, and the only thing worth proving first is that the intelligence runs in a loop a human will accept. Everything else is decoration on an unanswered question.

## The framework

Five decisions separate an experiment that teaches you something from a feature pile that flatters you.

1. **The Minimum Viable Agent.** A [Minimum Viable Agent](../dictionary/07-ai-native-business.md#minimum-viable-agent-mva) is the smallest agent that proves the intelligence loop closes: task in, agent acts, outcome out, human accepts or corrects, system records the choice and improves. A traditional MVP asks what is the least you can build that a user will pay for, and answers in features. The MVA asks a harder question, the least intelligence you can ship that proves a real human will hand a real job to a machine and keep the result. The deliverable is not a screen. It is a working loop around a single task, and it can be embarrassingly thin on the surface: a text box, an email reply, a row in a spreadsheet someone approves. The intelligence is the product. The interface is whatever is fastest to put the loop in front of a person.

2. **The Goldilocks zone.** Your first agent should own a task that is high-volume, guideline-based, and low-risk, all three at once. [The Goldilocks zone](../dictionary/07-ai-native-business.md#the-goldilocks-zone) is that profile. High-volume, so the loop runs often enough to generate signal and save time anyone cares about. Guideline-based, so there is a knowable right answer the agent can be measured against and a human can check at a glance, a rule or a rubric, not pure taste. Low-risk, so that when the agent is wrong, and early on it will be, the cost is an annoyance and not a recall. The temptation is always to point the first agent at the hardest, most valuable, most dangerous job, because that is the one that impresses the room. You have not earned that yet.

3. **Rapid prototyping loops.** A coding agent turns a day of customer feedback into a working iteration overnight, and that resets a clock you have lived under your whole career. The old cycle ran in sprints: learn something from a user, queue the change, wait for engineering, learn the next thing. The new cycle runs in sleeps. You sit with a pilot user at four in the afternoon, watch where the loop breaks, hand the synthesised feedback to a coding agent that evening, and walk into tomorrow's session with a version that fixes it. Change one thing per night. Five mornings with one isolated change each teaches you more than fifty customers seen once against a frozen build.

4. **Self-healing fallbacks.** A [self-healing fallback](../dictionary/04-agentic-patterns.md#self-healing-fallback) is the retry-and-reroute logic that keeps the loop alive when the primary model fails: the call times out, the model is overloaded, the agent returns garbage, and instead of the whole workflow face-planting, the system retries, switches to a backup, and keeps going. The failure this prevents is the one that quietly kills pilots. Your MVA sits behind an API, and that API will sometimes be slow or down or wrong. A pilot user who hits a dead end twice stops trusting the thing and goes back to doing it by hand. You did not lose them on capability. You lost them on reliability.

5. **Minimal viable experiment versus scope creep.** Test one high-risk hypothesis at a time instead of building a multi-feature product. This is hard precisely because AI made adding features frictionless, and a frictionless build-anything machine is a scope-creep engine with a friendly face. Building a feature feels like progress. Testing a hypothesis feels like risk, because the answer might come back no. So under uncertainty founders flee toward what feels safe and productive, more features, and away from the one experiment that could invalidate the company. Flip the unit of work from feature to experiment. If a proposed build does not map to a hypothesis that can fail, it is scope creep, however reasonable it sounds.

## Why it holds

Three companies, three ways of experimenting toward a product instead of building on faith. Weigh the move, not the digits.

**NotCo** runs its food R&D around a generative system that proposes alternative compositions, formulations that rebuild the taste and texture of an animal product from plant ingredients, by finding non-obvious patterns across molecular and sensory data. Traditional food R&D is a slow search through an astronomically large combinatorial space: a formulator guesses, the lab makes a batch, a panel tastes it, the result feeds the next guess, each turn costing days and money. NotCo used generative AI to de-risk the expensive front of that search. The agent owns the high-volume, pattern-finding part, proposing candidates worth physically making. Humans own the validation and the final call. That is the Goldilocks zone applied to deep R&D, and it is the move every founder whose moat is a search through a huge space should copy: make the human's search smarter and cheaper, do not remove the human from the decision that carries the risk.

**Beewise** builds AI-driven robotic hives that monitor colonies and intervene automatically to keep them healthy. The discipline worth studying is the order of operations. The company validated that the system actually reduced colony loss before scaling up hardware production. It reports cutting colony mortality against an industry baseline of roughly a third of colonies lost per year, down to a few percent for hives under its system [claimed in source]. A hardware company faces the most expensive version of the scope-creep trap, because manufacturing is capital, tooling, and inventory, a bet you cannot easily unwind. Beewise treated the central claim, does the monitor-and-intervene loop keep colonies alive, as a hypothesis to validate first, in the field, on real hives, while it was still cheap to be wrong. Verify the percentages before you repeat them. Copy the sequence regardless.

**Glass Health** is worth studying for what it chose as its surface. Clinical decision support is about as high-stakes as software gets, and the disciplined move is to enter through a narrow, guideline-based task rather than the open-ended job of practising medicine. A differential-diagnosis assistant that proposes structured possibilities for a clinician to review is a checkable loop with a human firmly on it. The agent does not own the decision. It owns the high-volume, rule-shaped front of the work, and the clinician keeps the judgement. That is the Goldilocks zone inside a domain where being wrong is dangerous: you find the adjacent task with a knowable answer and you earn trust there first.

## In hard mode

For a food, health, or deeptech founder the sequencing is not advice. It is survival.

Your highest-value job is almost always your highest-risk one. The contamination call. The dosing recommendation. The structural sign-off. That is precisely the job an agent should not own first, no matter how badly you want the headline, because the first wrong answer on a high-risk task does not just fail the task. It detonates the trust you need for the entire roadmap. People decide in that moment that the machine cannot be trusted, and they decide it about everything, not just the thing it got wrong.

So you pick a low-risk task inside the regulated workflow and earn trust there. Sort incoming lab results by which need urgent human review. Draft the first pass of a routine compliance form. Flag supplier documents missing a required field. None of these can harm someone if the agent slips, all of them run often enough to build a track record, and each one is checkable at a glance. The dangerous work stays [human-on-the-loop](../dictionary/04-agentic-patterns.md#human-on-the-loop-hotl) until the agent has a record that justifies loosening the leash.

Trust is earned in the order of stakes, not the order of value. A team that watches an agent triage two thousand low-stakes items week after week, with the occasional caught error and no disaster, slowly accumulates the one thing that lets them eventually hand it something that matters. Earn the right to point an agent at anything that can hurt a person. Do not seize it.

One more thing the hard-mode founder cannot skip: the fallback is part of the experiment, not a later polish. When confidence drops, or the question touches a dose or an allergen or a safety threshold, the loop has to fall to a fixed, non-AI path rather than guess fluently. Retry a bounded number of times. Reroute to a bounded set of backups. Then drop to a deterministic answer and escalate to a named human. Self-healing without a hard limit is just an infinite loop in a nicer coat, and recovery that never gives up does not heal, it burns tokens in a circle until someone notices the bill.

## What it means for you

1. **Write your MVA in one sentence.** Name the single task your first agent owns, then force it through all three Goldilocks tests. High-volume enough to save real time and throw off real signal. Guideline-based, with a knowable right answer a human can check at a glance. Low-risk, so a wrong answer is an annoyance and not a harm. Fail any one of the three and you have the wrong first task.
2. **Pick the safe adjacent job, not the valuable dangerous one.** If your highest-value task is also your highest-risk, write it on the roadmap for later and choose the boring task next to it now. You are buying a track record, and a track record is the only currency that buys you the dangerous job later.
3. **Spec the fallback before you build.** Decide what the agent does when it is unsure or fails, in one honest, bounded sentence: stops after a set number of retries, hands the task to a named human, tells the user the truth. That deterministic floor is what your whole MVA stands on.
4. **Run overnight loops, one change per night.** After each pilot session, hand the day's feedback to a coding agent that evening and change exactly one thing. Every cycle should answer a question you wrote down before you started. If you cannot name the question, you are not running an experiment, you are shipping at night.
5. **Hold every feature against the experiment.** Before you build anything, ask what you are most afraid to learn and what is the smallest build that forces you to learn it. State the trust hypothesis so it can come back no: users in role R, facing task T, will accept the agent's output without redoing it at least M percent of the time. If a no would save you a year, that is the build. The rest is scope creep. If the answer comes back a strong yes, that is the moment [the 40% rule](../dictionary/07-ai-native-business.md#the-40-rule-sean-ellis-test) becomes worth measuring.

## The test

Look at what you are building this week. Can you point to the single high-volume, guideline-based, low-risk task your first agent owns, and name the one trust hypothesis this build exists to test? If you are building features instead of running that one experiment, you are avoiding the question that decides whether you have a company. Run the [Remove-the-AI test](../dictionary/07-ai-native-business.md#the-remove-the-ai-test) on your MVA: take the model out, and if the loop still works, you scoped a feature, not an agent. Scope the experiment first. → Run `design-the-mva`.
