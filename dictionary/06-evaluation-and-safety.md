# Evaluation & Safety

Trust is not a final gate you bolt on before launch. It is a substrate you pour from day one, under everything else. This category is the engineering of that substrate: the tests, fallbacks, logs, and guardrails that let you replace human judgement with machine judgement one careful step at a time. Earn the right to remove the human; never just assume it.

### Eval

A repeatable test that measures the quality of a model or agent's output against a known-good expectation: the unit test of AI.

A normal unit test asserts `add(2, 2) === 4` and is either green or red. An eval is fuzzier because the output is fuzzier: it runs the same inputs through your system and scores the results: exact match, a rubric, a regex, or a [critic/judge](04-agentic-patterns.md#critic-judge-pairing) model grading against criteria. You run it on every prompt change, model swap, and [inference](01-the-model.md#inference) config tweak, and you watch the score move.

Without evals you are flying on vibes. You tweak a system prompt, the demo "feels better," you ship, and three weeks later a silent regression has been wrong the whole time. In health, food-safety, or finance, "feels better" is not a quality bar. An eval suite is the difference between a measured claim and a hopeful one. Build the eval before you build the feature; it defines what "working" even means.

_Avoid:_ thinking a passing demo is an eval. A demo is one lucky input observed once; an eval is many inputs scored repeatedly.

_Usage:_
> "Before we touch the dosage-calculation prompt, write the eval. I want a number to move, not a screenshot that looks nicer."

Related: [golden dataset](#golden-dataset), [critic/judge pairing](04-agentic-patterns.md#critic-judge-pairing), [verification deficit](05-failure-modes.md#verification-deficit)

### Golden dataset

A curated, version-controlled set of input/expected-output pairs that an [eval](#eval) runs against: the answer key for your system.

Each row is a representative input plus the output a domain expert agrees is correct. Some rows are the easy happy path; the valuable ones are the edge cases, the adversarial inputs, and the bugs you already fixed (so they can never silently return). The set grows every time production surprises you: a real failure becomes a new row, and the [eval](#eval) guarantees that specific failure stays dead.

The "golden" part is the discipline. Expected outputs are signed off by a human who knows the domain, a food-safety officer, a clinician, a compliance lead, not auto-generated by the same class of model you're testing, or you're just grading the model against itself. In regulated work, your golden dataset is also evidence: it shows an auditor exactly which cases you validated and what "correct" meant on the day you shipped.

_Avoid:_ calling a pile of raw production logs a golden dataset. Without verified expected outputs, it's a sample of inputs, not an answer key.

_Usage:_
> "Add the allergen-mislabel case to the golden dataset and freeze the expected output: that's the one that nearly shipped wrong last quarter."

Related: [eval](#eval), [versioned data](#versioned-data), [model drift](05-failure-modes.md#model-drift)

### Deterministic fallback

A non-AI, predictable code path that takes over when the model's confidence is low, its output fails validation, or it errors out entirely.

The model is the fast lane; the fallback is the road that's always open. When [uncertainty quantification](#uncertainty-quantification) drops below a threshold, a [guardrail](#guardrail) trips, or the API times out, the system stops asking the model and runs something boring and verifiable instead: a lookup table, a rules engine, a fixed safe message, or a clean handoff to a human. The behaviour is repeatable: same input, same output, every time, with no temperature involved.

This is the core of [human-on-the-loop](04-agentic-patterns.md#human-on-the-loop-hotl) safety. In an irreversible or sensitive action, releasing a batch, approving a transaction, giving a dosage, you never want "the model was having a bad day" to be the failure story. A good fallback turns a [silent failure](05-failure-modes.md#silent-failure) into a loud, predictable, safe one. Design it first, not after the incident: the fallback is what lets you ship the AI path at all.

_Avoid:_ confusing a fallback with a retry. Retrying the same flaky model is not a deterministic path. It's the same gamble, rolled twice.

_Usage:_
> "If confidence on the lab-result reading is under threshold, don't guess: drop to the deterministic fallback and flag it for a human."

Related: [uncertainty quantification](#uncertainty-quantification), [self-healing fallback](04-agentic-patterns.md#self-healing-fallback), [silent failure](05-failure-modes.md#silent-failure)

### Uncertainty quantification

A measure of how confident the system is in a given output, surfaced as a usable signal rather than buried inside the model.

A model will state a wrong answer with the exact same fluent tone as a right one. Confidence and correctness are not the same thing, and [hallucination](05-failure-modes.md#hallucination) is what happens when they diverge. Uncertainty quantification tries to recover a real confidence signal: token-level probabilities, agreement across multiple samples, a [critic/judge](04-agentic-patterns.md#critic-judge-pairing) second opinion, retrieval-coverage checks, or a calibrated self-estimate. The output is a number you can threshold on.

That number is what makes a [deterministic fallback](#deterministic-fallback) possible: you can only "drop to the safe path when unsure" if you can measure unsure. In sensitive domains, the rule is simple: low confidence must change behaviour. It should narrow the action, ask for more input, or route to a human, never just append a quiet "I might be wrong" to an answer the system ships anyway.

_Avoid:_ treating a model's self-reported "95% confident" as ground truth. Stated confidence is often poorly calibrated; verify it against actual hit rates before you trust the threshold.

_Usage:_
> "We can't act on this automatically until uncertainty quantification is wired in. I need a confidence score to gate the auto-approve."

Related: [deterministic fallback](#deterministic-fallback), [hallucination](05-failure-modes.md#hallucination), [temperature](01-the-model.md#temperature)

### Guardrail

A hard constraint that sits outside the model and blocks, filters, or reshapes its inputs and outputs regardless of what the model "wanted" to do.

Think of it as a fence, not advice. A system-prompt instruction is a suggestion the model can be talked out of; a guardrail is code that runs before the input reaches the model and after the output leaves it: input filters against [prompt injection](05-failure-modes.md#prompt-injection), output schema validation, banned-content checks, allow-lists of permitted tools and actions, rate and scope limits on what the agent can touch. If the model produces something outside the fence, the guardrail catches it; the model never gets a vote on whether the fence applies.

The discipline is to put guardrails in the runtime, not the prompt. Anything a clever input can argue the model out of was never a real control. In regulated work, guardrails encode the rules you cannot break, a financial agent that physically cannot move funds above a cap, a health agent that cannot output a diagnosis it's not cleared to give, and a tripped guardrail should route to a [deterministic fallback](#deterministic-fallback), not a quiet failure.

_Avoid:_ calling a politely worded system prompt a guardrail. If a clever input can talk the model out of it, it's a guideline, not a guardrail.

_Usage:_
> "Put the spend cap in a guardrail, not the prompt. I don't care how convincing the user's message is. The code refuses over 500 euro."

Related: [deterministic fallback](#deterministic-fallback), [prompt injection](05-failure-modes.md#prompt-injection), [red-teaming](#red-teaming)

### Red-teaming

The practice of deliberately attacking your own AI system to find the inputs and edge cases that make it fail before a real user or adversary does.

A normal [eval](#eval) checks that the system handles the inputs you expected. Red-teaming hunts for the inputs you didn't: [prompt injections](05-failure-modes.md#prompt-injection) buried in a document, jailbreaks that talk past a [guardrail](#guardrail), adversarial phrasings that coax out a confidently wrong answer, data that pushes the agent toward an unsafe action. You play the attacker, or dispatch an adversarial agent to do it, and you treat every success as a bug to fix, not a curiosity.

The payoff is that the failures you find in a red-team are cheap; the ones a regulator or a harmed user finds are not. Every confirmed exploit becomes a new row in the [golden dataset](#golden-dataset), so the gap closes permanently. In health, food-safety, and finance, red-teaming isn't optional polish: it's how you discover the failure mode that turns into a recall, a fine, or a headline.

_Avoid:_ confusing red-teaming with regular QA. QA confirms the happy path works; red-teaming assumes someone is actively trying to break it and goes looking for how.

_Usage:_
> "Red-team the intake agent before the pilot: try to get it to leak another patient's record. If you can, so can someone else."

Related: [guardrail](#guardrail), [prompt injection](05-failure-modes.md#prompt-injection), [golden dataset](#golden-dataset)

### Audit trail

An append-only record of what the system did, inputs, decisions, tool calls, outputs, and who or what triggered each step, kept so the past can be reconstructed exactly.

When an AI system makes a consequential call, "the model decided" is not an acceptable answer to "why did this happen?" An audit trail answers it: the prompt and context that went in, the model and version that ran, the confidence and [guardrail](#guardrail) checks, the action taken, the human who approved or was notified. Append-only and tamper-evident is the point: a log you can quietly edit after an incident proves nothing.

This is where compliance gets concrete. SOC 2, HIPAA, and ISO 27001 all require that activity be captured and reviewable, and here's the honest catch: some AI tools and agent frameworks don't record their actions in your audit logs at all: the work happens, but it's invisible to the system of record. For a regulated workload that's a real gap, not a detail. Before you put an AI agent anywhere near sensitive data, ask the unglamorous question: when the auditor asks what happened on this date, can I produce it?

_Avoid:_ treating ordinary application logging as an audit trail. Mutable logs that roll over in 30 days won't survive an audit or an incident review.

_Usage:_
> "Does the agent's tool call show up in our audit trail, or just in its own console? If it's not in the HIPAA log, we can't run it on patient data."

Related: [versioned data](#versioned-data), [the six-pillar trust substrate](#the-six-pillar-trust-substrate), [silent failure](05-failure-modes.md#silent-failure)

### Versioned data

The practice of treating datasets, prompts, and model configs as versioned artefacts, snapshotted, tagged, and reproducible, exactly the way you version code.

Code is versioned by reflex; the data and prompts that drive AI behaviour usually aren't, and that's the gap that bites. If your [golden dataset](#golden-dataset), your retrieval corpus, and your system prompt aren't pinned to versions, you can't answer the most basic incident question: what exactly was the system running when it gave that answer? Versioning means every output can be traced back to the precise data and prompt that produced it.

This is the antidote to a whole class of mystery bugs. [Model drift](05-failure-modes.md#model-drift) and a silently updated knowledge base look identical from the outside, both make yesterday's good answer today's bad one, and only versioning tells them apart. For regulated work it's also the backbone of reproducibility: a year from now you may need to recreate the exact state that produced a decision, and "we've changed the data since then" is not a defence.

_Avoid:_ confusing versioned data with a database backup. A backup restores the latest state; versioning lets you check out the exact state from any past decision.

_Usage:_
> "Tag the corpus version in the audit trail. If a result gets challenged in six months, I want to reload the exact data it was reading."

Related: [golden dataset](#golden-dataset), [audit trail](#audit-trail), [model drift](05-failure-modes.md#model-drift)

### The six-pillar trust substrate

A checklist of six foundations that an AI system handling consequential decisions in a sensitive domain must stand on: versioned data, immutable audit trails, uncertainty quantification, deterministic fallbacks, cybersecurity, and forensic auditability.

Read it as load-bearing pillars, not a menu. **[Versioned data](#versioned-data)** so every output traces to a known state. **Immutable [audit trails](#audit-trail)** so the past can't be quietly rewritten. **[Uncertainty quantification](#uncertainty-quantification)** so the system knows when it doesn't know. **[Deterministic fallbacks](#deterministic-fallback)** so low confidence triggers a safe, predictable path. **Cybersecurity** so the data and the model can't be tampered with from outside. **Forensic auditability** so that after an incident you can reconstruct exactly what happened, in what order, on whose authority.

The point of naming all six together is that they fail as a set. A system with perfect audit logs but no fallback still acts on bad guesses; one with great fallbacks but mutable logs can't prove what it did. In health, food-safety, or finance, missing any one pillar means you don't yet have the right to remove the human, so treat the six as the bar you clear before automating a consequential decision, not features you add later.

_Avoid:_ treating this as a launch-day checklist. The substrate is built into the architecture from the start; retrofitting trust onto a system that wasn't designed for it rarely holds.

_Usage:_
> "Before this agent touches the release decision, walk the six-pillar trust substrate. We're missing forensic auditability: that's a blocker, not a nice-to-have."

Related: [audit trail](#audit-trail), [deterministic fallback](#deterministic-fallback), [human-in-the-loop vs human-on-the-loop](#human-in-the-loop-vs-human-on-the-loop)

### Human-in-the-loop vs human-on-the-loop

Two distinct safety postures: in-the-loop means a human approves each action before it happens; on-the-loop means agents act autonomously while a human monitors and can intervene.

In-the-loop is a checkpoint: the agent proposes, the human disposes, nothing irreversible happens without a click. It's slow and it doesn't scale, but it's the right posture while you're still earning trust, and the only acceptable one for the most consequential actions. On-the-loop is a control room: agents run on their own, the human watches dashboards and alerts, and steps in when a [guardrail](#guardrail) trips or [uncertainty](#uncertainty-quantification) spikes. It scales, but only if the monitoring and the [deterministic fallbacks](#deterministic-fallback) underneath are real.

The whole arc of this category is about moving safely from one to the other: you start in-the-loop, build [evals](#eval), [audit trails](#audit-trail), and fallbacks until the system has earned it, then graduate specific low-risk actions to on-the-loop while keeping a human in-the-loop on the irreversible ones. The mistake is jumping straight to on-the-loop because it's faster: that's automation without the substrate to catch it when it's wrong.

_Avoid:_ using the two phrases interchangeably. The difference is whether a human approves *before* the action or reviews *after* it, and in irreversible domains, that difference is everything.

_Usage:_
> "Refunds under 50 euro can go human-on-the-loop now that the evals are solid. Anything that closes an account stays human-in-the-loop."
